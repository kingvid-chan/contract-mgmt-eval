# ADR-004: 数据库迁移策略 — 手动幂等 ALTER TABLE

**状态**: 已采纳  
**日期**: 2026-06-18  
**迭代**: 0.0.2

## 背景

项目使用 `Base.metadata.create_all()` 建表，没有 Alembic。0.0.2 迭代需要为 `audit_logs` 表新增 `ip_address` VARCHAR(45) 列。需选择迁移方式。

## 方案对比

| 方案 | 做法 | 优点 | 缺点 |
|------|------|------|------|
| **A: 手动幂等迁移** | `init_db()` 中执行 ALTER TABLE | 简单直接、保留数据 | 手动管理、后续迁移需继续手写 |
| B: 引入 Alembic | 正式迁移框架 | 规范、解决技术债 | 引入新依赖、初始化配置工作量大 |
| C: 重建表 | Drop + Create | 最简单 | 丢失已有审计日志数据 |

## 决策

选择 **方案 A — 手动幂等迁移函数**。

在 `database.py` 中新增 `_run_migrations()` 函数，在 `init_db()` 的 `create_all()` 之前执行：

```python
def _run_migrations():
    with engine.connect() as conn:
        # Migration 0.0.2: add ip_address column
        result = conn.execute(
            text("SELECT COUNT(*) FROM pragma_table_info('audit_logs') WHERE name='ip_address'")
        )
        if result.fetchone()[0] == 0:
            conn.execute(text("ALTER TABLE audit_logs ADD COLUMN ip_address VARCHAR(45)"))
            conn.commit()
```

幂等保证：多次执行不报错，列已存在则跳过。

## 影响

- `database.py` 新增 `_run_migrations()` 函数
- 后续迭代的新增列继续在此函数中追加
- 若字段变更复杂度增加（如重命名列、修改约束），再评估引入 Alembic
